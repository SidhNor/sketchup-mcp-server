# Step 01: Capture Problem

## Purpose

Establish a correct, shared understanding of the task before planning.

## Entry Criteria

- a defined task document exists
- the task links to at least one upstream source such as an HLD, PRD, or domain analysis

## Actions

1. Document the task problem statement in planning terms.
2. Identify affected components, systems, modules, runtimes, or boundaries.
3. List initial questions:
   - technical unknowns
   - business rule clarifications
   - edge cases
   - integration unknowns
4. Capture existing context from related tasks, HLDs, PRDs, domain analysis, and current repository structure where relevant.
5. Note any immediate conflicts or ambiguity.

## Expected Outputs

- problem summary
- affected component list
- initial question list
- related context summary with links or references

## Exit Criteria

- the problem is documented clearly enough that the user can confirm or correct it
- affected components are identified at a meaningful planning level
- open questions are explicit rather than implicit
- related context is linked or referenced

## Confirmation Gate

Present a concise summary of findings and ask the user for permission to proceed to Step 02.
