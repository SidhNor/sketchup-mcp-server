# Step 06: Architectural Context

## Purpose

Show how the planned work integrates with the surrounding system and where it must be tested.

## Entry Criteria

- Step 05 exit criteria are met
- the technical direction is confirmed or refined by the user

## Actions

1. Create a diagram showing:
   - involved components
   - integration points
   - runtime or process boundaries
   - test boundaries
2. Document key relationships between the affected components.
3. Highlight where real integration must be validated rather than assumed through mocks alone.
4. Keep the diagram simple and implementation-relevant.

## Expected Outputs

- architecture diagram
- key relationship summary
- integration-point summary
- test-boundary summary

## Exit Criteria

- implementers can see how the task fits into the surrounding system
- integration points are explicit
- test boundaries are explicit
- the diagram is clear enough to guide implementation and review

## Proceed Condition

Proceed to Step 07 when the integration context is understandable and aligned with the refined technical decisions.
