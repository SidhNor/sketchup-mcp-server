### Top Assumptions

- `MTA-02` terrain storage can persist the capped heightmap payloads generated by create/adopt without chunking, compression, or sidecars.
- Existing target resolution and surface sampling support can be reused for the supported adoption source class without a second raycast/sampling stack.
- The initial cap profile of `targetSamples: 4096`, `maxSamples: 10000`, `maxColumns: 128`, and `maxRows: 128` is acceptable for MTA-03 adoption quality and hosted performance.
- Source replacement can be committed atomically with owner creation, state save, derived output generation, and source erase in one SketchUp operation.
- No public adoption sampling policy, warnings, parent placement, or elevation-array create mode is required for MTA-03.

### Estimate Breakers

- If the supported adoption source cannot be sampled reliably through existing transformed/nested surface support, implementation friction and validation burden increase materially.
- If live SketchUp undo cannot restore source replacement and managed output coherently from one operation, the adoption design needs rework or source policy must change.
- If the 10,000-sample cap is too slow, too coarse, or too large for real payload/storage behavior, caps, refusals, docs, and tests must be revised.
- If creating the derived mesh from state requires broad geometry adapter or triangulation infrastructure beyond regular-grid output, technical surface expands.
- If clients require hierarchy-aware placement or public sampling controls in this slice, the public contract and test matrix widen.

### Predicted Signals

- Public tool addition requires synchronized loader schema, dispatcher/facade/factory routing, native contract fixtures, tests, README examples, and refusal documentation.
- The task combines product-visible create behavior, adoption/replacement behavior, persisted state, derived geometry output, and host undo semantics.
- Calibrated `MTA-02` shows terrain state/storage foundations carry high validation and rework sensitivity even without public tool behavior.
- Calibrated `STI-03` shows transformed/nested surface sampling and hosted visibility/ancestry behavior can expose implementation defects late.
- Calibrated `SVR-04` shows terrain sampling consumers need explicit hosted/performance evidence even when the public feature is bounded.

### Predicted Estimate Notes

- The predicted profile accounts for the pre-implementation planning rebaseline from adoption-only to create-or-adopt while preserving the original seed as historical baseline.
- Technical surface is scored very high because this is both a new public MCP tool and a stateful SketchUp geometry mutation workflow.
- Validation burden is the dominant cost driver; implementation is not complete without real or explicitly documented hosted coverage for create, adopt, undo, source replacement, and persistence.
- Confidence remains moderate-low because the exact supported source behavior and operation semantics must be proven in live SketchUp rather than inferred from local unit tests.

## Challenge Review

> Filled when the estimate is pressure-tested through external review, premortem, or controlled consensus.

### Agreed Drivers

- Public contract drift is a major estimate driver: the new tool must update loader schema, dispatcher/facade/factory routing, contract fixtures, README examples, and field descriptions together.
- Host-sensitive behavior is a dominant validation driver: source replacement, undo, transformed/nested sampling, save/reopen persistence, and derived mesh usability cannot be proven by local doubles alone.
- Adoption fidelity and cap behavior are material risks even without public sampling controls; evidence must report effective spacing, dimensions, sample count, cap, and practical hosted performance.
- The scope boundary must stay tight: no broad TIN import, mesh repair, public sampling policy, parent placement, hardscape mutation, or elevation-array create mode.

### Contested Drivers

- Isolated Grok 4.20 premortem challenged the separate public tool boundary and suggested folding terrain creation into `create_site_element`; planning adjudication rejects that as a blocker because managed terrain owns repository-backed terrain state, source replacement, and derived terrain output, which are distinct from semantic site-element creation.
- Grok challenged the response envelope for not using the MCP guide's preferred `status` vocabulary; planning preserves the existing project `ToolResponse.success` / `outcome` convention but now carries explicit contract-doc and fixture guardrails.
- Grok challenged the lack of a `hosting` section for adoption semantics; planning keeps adoption source selection in `lifecycle.target` because the guide says adoption/replacement targets belong in `lifecycle`, while terrain conformity remains out of scope for MTA-03.
- Grok's concern about underspecified supported source classes was accepted and converted into stricter plan language and tests for supported/unsupported adoption sources.

### Missing Evidence

- Hosted evidence that the supported source class can be sampled completely and unambiguously in realistic transformed/nested SketchUp contexts.
- Hosted or documented manual evidence that one SketchUp operation makes adoption undo restore source geometry and remove the managed replacement coherently.
- Practical performance evidence for representative capped adoption, including the 40m by 70m case and the 10,000-sample cap.
- Contract-readability evidence from loader descriptions, README examples, and native fixtures that `create_terrain_surface` does not overlap ambiguously with `create_site_element`.

### Recommendation

- Confirm the challenged estimate without score changes.
- Do not split the task before implementation; the premortem changes add guardrails and validation obligations, not a new product slice.
- Revisit split/defer only if hosted adoption cannot meet source replacement, undo, or cap-performance expectations.

### Challenge Notes

- No predicted score changed because the original profile already rated technical surface `4`, validation burden `4`, volatility `3`, rework risk `3`, and confidence `2`.
- The challenged review strengthens the plan through explicit boundary justification, stricter supported-source language, deterministic mesh requirements, and hosted performance/undo validation.
- The isolated Grok 4.20 premortem was used as challenge evidence without exposing it to prior premortem conclusions.

- Functional scope matched prediction: the public create/adopt vertical slice shipped without broad terrain import or edit tools.

Live-host geometric sampling semantics were the dominant source of rework: public unit conversion, boundary classification, and sampling candidate filtering all had to be proven against real SketchUp behavior rather than local assumptions.
